# **Spring Core**

## 1. IoC (Inversion of Control)
-   `IOC is a concept where control of object creation & dependency management is given to Spring container, instead of we creating mannually using new keyword.`

    ### Why IOC?
    - Reduces mannual object creation.
    - Makes code loosely coupled.
    - Spring manages bean lifecycle.

    ### How IOC works?
    - Spring scans annotated classes.
    - Creates beans.
    - Stores them inside the container.
    - Injects them whenever required. (Using `@Autowired`)


## 2. IoC Container

-   `IoC container is the component that implements IOC concept by creating and managing beans.`
- **IoC container is the general term.**<br>
`BeanFactory and ApplicationContext are actual implementations of that container in Spring Framework.`

* **What IOC Container does:**
    - `Creates Beans`
    - `Performs Dependency Injection`
    - `Manages their lifecycle`

-   Types:
    -   BeanFactory
    -   ApplicationContext
    ```
    IoC Container (concept)
      │
      ├── BeanFactory
      │
      └── ApplicationContext
    ```

    | Feature      | BeanFactory     | ApplicationContext |
    | ------------ | --------------- | ------------------ |
    | Type         | Basic container | Advanced container |
    | Bean loading | Lazy loading    | Eager loading      |
    | Features     | Limited         | More features      |
    | Common usage | Rarely used     | Most commonly used |

* **`Lazy`** - **Means beans are created when requested.**
    

## 3. Dependency Injection (DI)
* `DI is a design pattern where Spring creates and provides the dependencies for the objects, instead of we creating manually using new keyword.`
* `Design pattern:` **A design pattern is a blueprint that solves commonly occurring problems in software design.**

    1. **Object creation problems:**
        * **👉 “How to create objects efficiently?”**
            - `Too many objects ❌`
            - `Complex creation ❌`
        * **✔️ Solution → Singleton, Factory**

    2. **Dependency management problems**
        * **👉 “How to manage dependencies between objects?”**
            - `Tight coupling (new keyword everywhere) ❌`
            - `Hard to test (no mocking) ❌`
        * **✔️ Solution → Dependency Injection (Constructor Injection, Setter Injection)**

* **Ways Spring injects dependencies:** 
    - Setter Injection 
    - Constructor Injection 
    - Field Injection

    ### 👉 Why Constructor Injection is preferred -

    1. **Mandatory dependencies:**
        * `Bean cannot be created without required dependencies.`
            ```
            @Component
            class Car {

                private Engine engine;

                public Car(Engine engine) {
                    this.engine = engine;
                }
            }
            ```

    2. **Immutable objects:**
        * `We can make dependencies final, so they cannot change later.`

    3. **No NullPointerException risk:**
        * `Bean always gets its dependency → less chance of null`


    ### 👉 Why we cannot give `final` to `Setter Injection ` and `Field Injection`?
    * **Setter Injection and Field Injection:**
        - Happens after object creation
        - but **`final`** variables must be initialized during object creation
        - Therefore final cannot be used with **`Setter Injection`** & **`Field Injection`**

    * **Constructor Injection:**
        - Happens during object creation
        - **`final`** variables can be initialized inside the constructor
        - Therefore final works with **`constructor injection`**

## 5. Bean Scope
#### 👉 `Bean scope means how many beans should spring create and how long they live.`
* **Common scopes:**
    - **`singleton`**  → Only **one bean instance** per Spring container (default) 
    - **`prototype`**  →  **New bean instance** every time it is requested        
    - **`request`**    →  One bean instance per **HTTP request**                  
    - **`session`**    →  One bean instance per **HTTP session**                  
    - **`application`** → One bean instance per **ServletContext (application)**  
    - **`websocket`**   → One bean instance per **WebSocket session**             

* **Stateless:** `Stateless means it does not store data in instance variables.`
    - **singleton is used for stateless beans.**
    - **We have to design them to be stateless.**

* **Example:** `<bean id="student" class="Student" scope="prototype"/>`

## 6. Bean Configuration
#### 👉 `Procedure of defining and creating beans so that Spring IoC container can create and manage them.`
* Ways to define beans: 
    1. **XML config**
    2. **Annotation config** 
    3. **Java config (`@Configuration`, `@Bean`)** <br><br>

    | Type              | Configuration Happens in      | Container reads |
    | ----------------- | ----------------------------- | --------------- |
    | XML Config        | XML file                      | XML             |
    | Annotation Config | Annotations + XML             | Annotations     |
    | Java Config       | Java class (`@Configuration`) | Java class      |


    ### **1. XML config**
    * **We can perform `Dependency Injection` using `XML configuration` in 2 ways.**
        1. **Setter Injection -**
            * Value Injection:
                ``` 
                <bean id="student" class="Student">
                    <property name="name" value="Rahul"></property>
                </bean>

                This injects primitive/String value.
                ```

            * Object Injection:
                ```
                <bean id="student" class="Student">
                    <property name="address" ref="gcAddress"></property>
                </bean>

                This injects another bean object.
                ```  

        2. **Constructor Injection -**
            * Value Injection:
                ```
                <bean id="student" class="Student">
                    <constructor-arg name="name" value="Rahul"></constructor-arg>
                </bean>
                ```

            * Object Injection:
                ```
                <bean id="student" class="Student">
                    <constructor-arg name="address" ref="gcAddress"></constructor-arg>
                </bean>
                ```

## 9. Bean Lifecycle
* **Bean Lifecycle Steps:**
    1. **Bean Instantiation (Creation)**
        - `Spring creates the bean.`

    2. **Dependency Injection**
        - `Spring injects required dependencies into that bean.`

    3. **Initialization**
        - `Methods like @PostConstruct run.`

    4. **Bean Ready for Use**

    5. **Destruction**
        - `@PreDestroy runs when the container shuts down.`

* **Common methods/annotations:**
    - `init()` 
    - `destroy()` 
    - `@PostConstruct` 
    - `@PreDestroy`

    ```
    @PostConstruct
    public void init() {
        System.out.println("Car is now ready");
    }
    ```
    ```
    @PreDestroy
    public void destroy() {
        System.out.println("Closing the resouces before car is destroyed");
    }
    ```

## 11. Difference between @Component and @Bean

| Feature                   | `@Component` | `@Bean` |
| ------------------------- | ------------ | ------- |
| Bean creation             | Automatic    | Manual  |
| Used on                   | Class        | Method  |
| Requires control of class | Yes          | No      |
| Third-party classes       | ❌            | ✅       |

* **`@Component`** - `automatically registered the class as a bean using component scanning`
* **`@Bean`** - `inside a @Configuration class is used to manually define how a bean should be created, especially for third-party or complex objects.`
